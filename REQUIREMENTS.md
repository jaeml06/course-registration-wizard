# Course Registration Wizard 요구사항 원천 문서

이 문서는 `.codex/prompts/specify.md`로 명세를 만들 때 읽어야 하는 과제 요구사항의 원천이다. `spec.md`를 생성할 때 이 문서의 요구사항을 축소하거나 임의로 제외하지 않는다.

## 1. 과제 목적

온라인 교육 플랫폼에서 강의 수강 신청 흐름을 3단계 다단계 폼으로 구현한다.

평가의 중심은 화면 장식이 아니라 다음 항목이다.

- 폼 상태 관리
- 스텝별 유효성 검증
- 개인/단체 신청 조건부 필드 처리
- 스텝 간 데이터 유지
- 제출 실패 후 데이터 보존과 재시도 UX

## 2. 핵심 사용자 흐름

```text
1단계 강의 선택
  -> 2단계 수강생 정보 입력
  -> 3단계 확인 및 약관 동의
  -> 제출 성공 화면 또는 제출 실패 후 재시도
```

사용자는 이전 단계로 돌아가도 입력한 정보를 잃지 않아야 한다.

## 3. 필수 구현 범위

### 3.1 1단계: 강의 선택

- 사용자는 카테고리별 강의 목록을 조회할 수 있어야 한다.
- 사용자는 수강할 강의를 하나 선택할 수 있어야 한다.
- 강의 목록에는 강의 제목, 설명, 가격, 일정, 강사, 정원 상태가 표시되어야 한다.
- 선택한 강의의 제목, 가격, 일정 등 주요 정보가 별도로 표시되어야 한다.
- 사용자는 신청 유형을 개인 신청 또는 단체 신청 중 하나로 선택할 수 있어야 한다.
- 정원이 찬 강의는 선택할 수 없어야 한다.
- 남은 정원이 적은 강의는 사용자가 즉시 알 수 있게 표시해야 한다.
- 강의 목록 조회의 로딩, 빈 목록, 조회 실패 상태를 각각 보여줘야 한다.

### 3.2 2단계: 수강생 정보 입력

공통 필드는 개인/단체 신청 모두에 필요하다.

| 필드 | 필수 여부 | 검증 규칙 |
| --- | --- | --- |
| 이름 | 필수 | 2~20자 |
| 이메일 | 필수 | 이메일 형식 |
| 전화번호 | 필수 | 한국 전화번호 형식 |
| 수강 동기 | 선택 | 최대 300자 |

단체 신청 시 추가 필드가 필요하다.

| 필드 | 필수 여부 | 검증 규칙 |
| --- | --- | --- |
| 단체명 | 필수 | 비어 있으면 안 됨 |
| 신청 인원수 | 필수 | 2~10명 |
| 참가자 명단 | 필수 | 신청 인원수와 정확히 같은 개수 |
| 참가자 이름 | 필수 | 비어 있으면 안 됨 |
| 참가자 이메일 | 필수 | 이메일 형식, 중복 불가 |
| 담당자 연락처 | 필수 | 한국 전화번호 형식 |

### 3.3 3단계: 확인 및 제출

- 1~2단계에서 입력한 전체 내용을 읽기 쉬운 요약으로 보여줘야 한다.
- 강의 선택 섹션과 수강생 정보 섹션에는 각각 수정 링크가 있어야 한다.
- 수정 링크를 누르면 해당 스텝으로 이동하고 기존 입력 데이터는 유지되어야 한다.
- 이용약관 동의 체크박스가 있어야 한다.
- 약관에 동의하지 않으면 제출할 수 없어야 한다.
- 제출 중에는 제출 버튼을 비활성화하거나 요청 잠금을 사용해 중복 제출을 막아야 한다.

### 3.4 제출 결과

- 성공 시 신청 완료 화면을 보여준다.
- 성공 화면에는 신청 번호, 신청 상태, 신청 시각, 신청 요약 정보가 포함되어야 한다.
- 실패 시 사용자가 이해할 수 있는 에러 메시지를 보여준다.
- 실패 후에도 모든 입력 데이터는 유지되어야 하고 사용자는 다시 제출할 수 있어야 한다.

## 4. 조건부 필드 데이터 정책

단체 신청은 데이터 정합성 이슈가 많으므로 아래 정책을 반드시 따른다.

| 상황 | 처리 정책 |
| --- | --- |
| 개인 신청 -> 단체 신청 | 단체 필드를 초기 기본값으로 생성 |
| 단체 신청 -> 개인 신청, 입력된 단체 정보 없음 | 단체 데이터를 초기화하고 개인 신청으로 변경 |
| 단체 신청 -> 개인 신청, 입력된 단체 정보 있음 | 사용자 확인 후 단체 데이터를 초기화 |
| 개인 신청 제출 | `group` 데이터는 제출 payload에서 제외 |
| 단체 신청 제출 | `group` 데이터는 제출 payload에 포함 |

참가자 이메일 정책:

- 참가자 이메일은 서로 중복될 수 없다.
- 참가자 이메일과 신청자 이메일도 중복될 수 없다.
- 다른 정책을 선택해야 한다면 README의 "요구사항 해석 및 가정"에 이유를 남긴다.

정원 정책:

- 선택한 강의의 잔여 정원은 `maxCapacity - currentEnrollment`로 계산한다.
- 단체 신청 인원수가 잔여 정원보다 크면 다음 단계 이동 또는 제출 전에 막고 이유를 설명한다.

## 5. 유효성 검증 UX

- 각 스텝에서 다음 단계로 넘어가기 전에 해당 스텝만 검증한다.
- 필드 blur 시 개별 검증을 제공하는 것을 우선한다.
- 검증 실패 시 필드별 에러 메시지를 보여준다.
- 검증 실패 필드는 시각적으로 강조한다.
- 가능한 경우 첫 에러 필드로 포커스를 이동하거나 스크롤한다.
- 서버 측 `INVALID_INPUT`의 `details`는 가능한 필드별 에러로 매핑한다.

## 6. Mock API 스키마

### 6.1 강의 목록 조회

```ts
GET /api/courses?category={category}

interface Course {
  id: string;
  title: string;
  description: string;
  category: string; // "development" | "design" | "marketing" | "business"
  price: number;
  maxCapacity: number;
  currentEnrollment: number;
  startDate: string; // ISO 8601
  endDate: string; // ISO 8601
  instructor: string;
}

interface CourseListResponse {
  courses: Course[];
  categories: string[];
}
```

### 6.2 수강 신청 제출

```ts
POST /api/enrollments

interface PersonalEnrollmentRequest {
  courseId: string;
  type: "personal";
  applicant: {
    name: string;
    email: string;
    phone: string;
    motivation?: string;
  };
  agreedToTerms: boolean;
}

interface GroupEnrollmentRequest {
  courseId: string;
  type: "group";
  applicant: {
    name: string;
    email: string;
    phone: string;
    motivation?: string;
  };
  group: {
    organizationName: string;
    headCount: number;
    participants: Array<{ name: string; email: string }>;
    contactPerson: string;
  };
  agreedToTerms: boolean;
}

interface EnrollmentResponse {
  enrollmentId: string;
  status: "confirmed" | "pending";
  enrolledAt: string;
}
```

### 6.3 에러 응답

```ts
interface ErrorResponse {
  code: string;
  message: string;
  details?: Record<string, string>;
}
```

서버 에러 코드는 사용자에게 이해 가능한 한국어 메시지로 변환한다.

| 코드 | 의미 | 사용자 메시지 방향 |
| --- | --- | --- |
| `COURSE_FULL` | 정원 초과 | 선택한 강의의 정원이 마감되었음을 알리고 강의 선택 수정을 유도 |
| `DUPLICATE_ENROLLMENT` | 이미 신청된 강의 | 이미 신청된 강의임을 알리고 재확인 유도 |
| `INVALID_INPUT` | 서버 측 입력값 오류 | 필드별 오류가 있으면 해당 필드에 매핑 |

## 7. 선택 구현

아래 항목은 추가 점수 영역이다. 핵심 기능을 먼저 안정화한 뒤 구현한다.

- 임시 저장: 새로고침 후에도 입력 데이터 복구
- 이탈 방지: 입력 중 브라우저 뒤로가기 또는 닫기 시 확인 대화상자
- 반응형 레이아웃: 모바일에서 스텝별 세로 스크롤 레이아웃

## 8. 기술 제약과 권장 방향

- TypeScript는 필수다.
- React + Vite를 기본 선택지로 둔다.
- 폼 상태는 통합된 단일 상태로 관리한다.
- 개인/단체 신청 타입은 discriminated union으로 분리한다.
- 유효성 검증 로직은 UI 컴포넌트에서 분리한다.
- API 타입, Mock 핸들러, 유효성 검증 유틸, 폼 컴포넌트를 역할별로 나눈다.
- 라이브러리를 추가하면 README의 "기술 스택"과 "설계 결정과 이유"에 선택 이유를 적는다.

## 9. 테스트 우선순위

구현 시 다음 영역을 우선 테스트한다.

- 이름, 이메일, 한국 전화번호, 수강 동기 길이 검증
- 개인/단체 신청 전환 시 데이터 처리
- 단체 참가자 수와 명단 개수 동기화
- 참가자 이메일 중복 검증
- 신청자 이메일과 참가자 이메일 중복 검증
- 단계 전환 검증
- 제출 성공 처리
- `COURSE_FULL`, `DUPLICATE_ENROLLMENT`, `INVALID_INPUT` 처리

## 11. specify 명세 생성 시 고정해야 할 사용자 스토리

`specify` 단계에서는 최소한 아래 사용자 스토리 축을 유지한다.

| 우선순위 | 사용자 스토리 |
| --- | --- |
| P1 | 수강생이 강의를 선택하고 개인 신청으로 제출할 수 있다 |
| P1 | 수강생이 단체 신청 정보를 정확히 입력하고 제출할 수 있다 |
| P1 | 스텝 이동 시 현재 스텝 검증과 입력 데이터 유지가 동작한다 |
| P2 | 정원, 빈 목록, 로딩, 조회 실패, 서버 에러를 이해 가능한 UX로 처리한다 |
| P2 | 확인 화면에서 전체 내용을 검토하고 섹션별로 수정할 수 있다 |
| P3 | 임시 저장, 이탈 방지, 반응형 레이아웃을 제공한다 |

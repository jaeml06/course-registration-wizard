# Course Registration Wizard 작업 지침

이 저장소는 온라인 교육 플랫폼의 **다단계 수강 신청 폼** 과제를 구현하기 위한 공간이다. 모든 AI 에이전트는 아래 지침을 우선해서 따른다.

## 답변 스타일

- 한국어로 친절하고 자세히 설명한다.
- 초보자도 따라올 수 있도록 필요하면 예시, 간단한 표, 흐름도를 함께 쓴다.
- 구현 결과를 말할 때는 “무엇을 바꿨는지”, “왜 그렇게 했는지”, “어떻게 확인했는지”를 구분해서 설명한다.

## 과제 핵심 목표

구현의 중심은 예쁜 화면보다 **폼 상태 관리, 단계별 유효성 검증, 조건부 필드, 실패 복구 UX**다.

반드시 고려할 사용자 흐름:

```text
1단계 강의 선택
  ↓
2단계 수강생 정보 입력
  ↓
3단계 확인 및 약관 동의
  ↓
제출 성공 또는 실패 후 재시도
```

## 문서와 작업 흐름

이 저장소는 `.codex/prompts/specify.md`에서 시작하는 Spec Kit 흐름을 기본 작업 방식으로 사용한다.

```text
specify
  -> clarify/checklist
  -> plan
  -> tasks
  -> analyze
  -> implement
  -> lint/test/README 갱신
```

문서 역할은 다음과 같이 구분한다.

| 문서 | 역할 |
| --- | --- |
| `REQUIREMENTS.md` | 과제 요구사항 원천 문서. `specify`가 반드시 읽어야 한다. |
| `AGENTS.md` | AI 에이전트의 작업 규칙과 우선순위. |
| `.specify/memory/constitution.md` | 코드 구조, TDD, 품질 게이트 같은 구현 원칙. |
| `ACCEPTANCE_CHECKLIST.md` | specify/plan/tasks/implement 단계별 누락 방지 체크리스트. |
| `README.md` | 최종 제출용 설명 문서. 구현 후 최신 상태로 갱신한다. |

우선순위는 `AGENTS.md`와 `REQUIREMENTS.md`를 먼저 따르고, 구현 세부 규칙은 `.specify/memory/constitution.md`를 따른다. 서로 충돌하면 사용자에게 질문하거나, 합리적 가정을 README의 “요구사항 해석 및 가정”에 남긴다.

`specify` 단계에서는 `REQUIREMENTS.md`의 기능 요구사항을 축소하지 않는다. 다만 `spec.md` 자체는 사용자가 무엇을 할 수 있어야 하는지와 왜 필요한지에 집중하고, React/Vite 같은 구현 기술 선택은 `plan.md`에서 구체화한다.

## Git 작업 정책

이 과제는 별도 기능 브랜치를 만들지 않고 `main` 브랜치에서만 작업한다.

- `specify`, `plan`, `tasks`, `implement` 모든 단계는 현재 `main` 브랜치를 유지한다.
- AI 에이전트는 사용자가 명시적으로 요청하지 않는 한 `git checkout`, `git switch`, `git pull`, `git push`, PR 생성 명령을 실행하지 않는다.
- Spec Kit 산출물은 브랜치 이름이 아니라 `specs/{type}/{NNN}-{slug}/` 디렉터리로 구분한다.
- `.specify` 스크립트가 feature branch를 요구하는 경우, `SPECIFY_FEATURE` 또는 가장 최근 spec 디렉터리를 사용해 main-only 흐름으로 진행한다.
- 완료 보고에서는 “생성한 브랜치”가 아니라 “작업 브랜치: main”과 “spec 경로”를 보고한다.

## 필수 기능 기준

- 3단계 폼을 구현한다.
  - 1단계: 카테고리별 강의 목록, 강의 선택, 선택 강의 정보, 개인/단체 신청 유형.
  - 2단계: 신청자 공통 정보, 단체 신청 조건부 필드.
  - 3단계: 전체 요약, 섹션별 수정 링크, 약관 동의, 제출.
- 이전 단계로 돌아가도 입력 데이터가 유지되어야 한다.
- 각 단계에서 다음 단계로 넘어가기 전에 해당 단계만 먼저 검증한다.
- 필드 blur 시 개별 검증을 제공하는 것을 우선한다.
- 검증 실패 시 필드별 에러 메시지, 시각적 강조, 가능한 경우 첫 에러 필드 포커스 이동을 제공한다.
- 제출 실패 시 입력 데이터는 유지하고 재시도 가능해야 한다.
- 중복 제출을 막기 위해 제출 중 버튼 비활성화 또는 요청 잠금을 둔다.

## specify 단계 고정 요구사항

`.codex/prompts/specify.md`를 사용할 때는 아래 항목을 반드시 명세에 반영한다.

- 3단계 흐름: 강의 선택 → 수강생 정보 입력 → 확인 및 제출.
- 개인 신청과 단체 신청을 각각 독립적으로 테스트 가능한 사용자 스토리로 작성한다.
- 스텝 이동 전 현재 스텝 검증과 이전 단계 데이터 유지 정책을 명시한다.
- 단체 신청 조건부 필드 전환 정책을 명시한다.
- 정원 마감, 잔여 정원 부족, 강의 목록 로딩/빈 상태/실패 상태를 명시한다.
- 제출 성공, 제출 실패, 서버 에러 코드별 복구 UX를 명시한다.
- 인증, 결제, 관리자 기능은 scope 밖으로 둔다.

이미 이 문서와 `REQUIREMENTS.md`에 명확한 정책은 clarification 질문으로 다시 묻지 않는다.

## 조건부 필드 정책

단체 신청은 개인 신청보다 데이터 정합성 이슈가 많으므로 아래 정책을 명시적으로 따른다.

- 개인 신청에서 단체 신청으로 바꾸면 단체 필드를 초기 기본값으로 생성한다.
- 단체 신청에서 개인 신청으로 바꾸면 단체 데이터는 제출 대상에서 제외한다.
- 이미 단체 정보가 입력된 상태에서 개인 신청으로 바꾸려는 경우, 사용자에게 확인을 받은 뒤 단체 데이터를 초기화한다.
- 신청 인원수는 2~10명이다.
- 참가자 명단은 신청 인원수와 정확히 같은 개수여야 한다.
- 참가자 이메일은 서로 중복되면 안 된다.
- 참가자 이메일과 신청자 이메일이 중복되는 경우는 허용하지 않는 방향을 기본값으로 한다. 다른 정책을 쓰려면 README의 “요구사항 해석 및 가정”에 이유를 남긴다.

## 정원과 강의 선택 UX

- 정원이 찬 강의는 선택할 수 없게 처리한다.
- 남은 정원이 적은 강의는 사용자에게 명확히 보여준다.
- 단체 신청 인원수가 선택한 강의의 잔여 정원보다 크면 다음 단계 또는 제출 전에 막고 설명한다.
- 빈 강의 목록, 로딩, 조회 실패 상태를 각각 보여준다.

## API Mock 기준

과제 명세의 API 스키마를 기준으로 Mock 환경을 구성한다.

- `GET /api/courses?category={category}`
- `POST /api/enrollments`

서버 에러 코드는 사용자에게 이해 가능한 메시지로 바꿔 보여준다.

- `COURSE_FULL`: 정원 초과
- `DUPLICATE_ENROLLMENT`: 이미 신청된 강의
- `INVALID_INPUT`: 서버 측 입력값 검증 실패

## 권장 기술 방향

- TypeScript는 필수다.
- React + Vite를 기본 선택지로 둔다.
- 폼 상태는 통합된 단일 상태로 관리한다.
- 개인/단체 신청 타입은 discriminated union으로 분리한다.
- 유효성 검증 로직은 UI 컴포넌트에서 분리한다.
- API 타입, Mock 핸들러, 유효성 검증 유틸, 폼 컴포넌트를 역할별로 나눈다.
- 라이브러리를 추가하면 README의 “기술 스택”과 “설계 결정과 이유”에 선택 이유를 적는다.

## 테스트와 검증

구현 시 다음 영역을 우선 테스트한다.

- 이름, 이메일, 한국 전화번호, 수강 동기 길이 검증.
- 개인/단체 신청 전환 시 데이터 처리.
- 단체 참가자 수와 명단 개수 동기화.
- 참가자 이메일 중복 검증.
- 단계 전환 검증.
- 제출 성공, `COURSE_FULL`, `DUPLICATE_ENROLLMENT`, `INVALID_INPUT` 처리.

가능하면 다음 명령으로 검증한다.

```bash
npm run lint
npm run test
```

프로젝트가 아직 초기화되지 않아 명령이 없으면, 먼저 그 사실을 보고하고 package scripts를 만든 뒤 다시 검증한다.

## README 작성 기준

최종 제출 전 README에는 과제 명세의 필수 항목을 모두 포함한다.

- 프로젝트 개요
- 기술 스택
- 실행 방법
- 프로젝트 구조 설명
- 요구사항 해석 및 가정
- 설계 결정과 이유
- 미구현 / 제약사항
- AI 활용 범위

## 작업 원칙

- 과제 요구사항과 평가 기준에 직접 연결되는 변경을 우선한다.
- 불필요한 인증, 결제, 관리자 기능, 과한 디자인 시스템은 만들지 않는다.
- 구현 전에 애매한 공개 동작이 있으면 질문하거나, 합리적 가정을 README에 남긴다.
- 기존 사용자의 변경사항은 되돌리지 않는다.
